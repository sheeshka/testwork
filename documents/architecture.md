## Депозит — создание платежа

```mermaid
flowchart TD
    A[POST /payments/deposit] --> B{Заказ существует?}
    B -- Нет --> C[404 Not Found]
    B -- Да --> D{Сумма платежей + новый <= сумма заказа?}
    D -- Нет --> E[400 Сумма превышена]
    D -- Да --> F{Тип платежа}
    F -- cash --> G[Создать платёж, status=completed]
    G --> H[Пересчитать payment_status заказа]
    F -- acquiring --> I[BankClient.acquiring_start]
    I -- Ошибка --> J[502 Ошибка банка]
    I -- Успех --> K[Создать платёж, status=pending]
    K --> H
    H --> L[Вернуть платёж]
```


## Статусы

### Статус оплаты заказа (PaymentStatus)

| Значение | Описание |
| --- | --- |
| `not_paid` | Не оплачен — нет завершённых платежей |
| `partially_paid` | Частично оплачен — сумма платежей меньше суммы заказа |
| `paid` | Оплачен — сумма платежей равна сумме заказа |

### Тип платежа (PaymentType)

| Значение | Описание |
| --- | --- |
| `cash` | Наличные — платёж завершается сразу |
| `acquiring` | Эквайринг — платёж через банк, требует подтверждения |

### Статус платежа (TransactionStatus)

| Значение | Описание |
| --- | --- |
| `pending` | Ожидает подтверждения от банка |
| `completed` | Завершён — деньги получены |
| `refunded` | Возвращён |
| `failed` | Ошибка — платёж не прошёл |

## Пересчёт статуса заказа

```mermaid
flowchart TD
    A[Сумма completed-платежей] --> B{= 0?}
    B -- Да --> C[not_paid]
    B -- Нет --> D{= сумма заказа?}
    D -- Да --> E[paid]
    D -- Нет --> F[partially_paid]
```

## Архитектурные решения

### Unit of Work

Все операции с БД проходят через UoW — это гарантирует атомарность: если при создании платежа не удалось обновить статус заказа, откатится вся транзакция. В `_repo_map` регистрируются репозитории, которые автоматически получают сессию при входе в контекст.

### Единая модель платежа

По требованию ТЗ модели платежей разных типов не отличаются друг от друга. Тип хранится в поле `type` (cash/acquiring), а логика различается только в сервисе: cash-платёж завершается сразу, acquiring — требует подтверждения от банка. Это позволяет работать с платежами единообразно в репозиториях, API и при пересчёте статуса заказа.

### RESTRICT вместо CASCADE

FK `payments.order_id` → `orders.id` использует `ON DELETE RESTRICT`. Удаление заказа при наличии платежей запрещено — это защита целостности финансовых данных. Операция удаления заказов не предусмотрена ТЗ.

### Явная синхронизация статуса

Банк — внешняя система, состояние платежа может измениться в любой момент. Эндпоинт `POST /payments/{id}/sync` позволяет явно запросить актуальный статус. Это проще и надёжнее, чем фоновая синхронизация, и достаточно для демонстрации архитектурного решения.

### BankClient — изоляция внешней зависимости

HTTP-клиент банка вынесен в отдельный модуль `clients/bank.py` с собственным типом ошибки `BankClientError`. Сервис не знает деталей HTTP-взаимодействия, а API-слой транслирует `BankClientError` в HTTP 502. Это упрощает замену клиента (mock для тестов, другой банк) без изменения бизнес-логики.
